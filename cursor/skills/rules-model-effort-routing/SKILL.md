# Выбор модели и маршрутизация (Cursor)

## Свобода выбора модели — не зашивать конкретную

В Cursor модель выбирается **свободно** из доступных в аккаунте (`cursor-agent models`). По умолчанию `auto` адаптируется сам — этого достаточно для большинства задач. **Не пиннить конкретную модель без причины.** Правило «зашивать модель» из Claude Code сюда НЕ переносится: там выбора не было, здесь он есть — пользоваться свободой.

Когда всё же нужно направить выбор — думать **tier'ами способность×стоимость**, а не именами моделей:
- Механическое / поиск / lookup / простой CRUD → лёгкая, быстрая модель.
- Ограниченная реализация (код, рефакторинг, code review, ручное QA, build engineering) → средняя.
- Сложный reasoning (планирование, архитектура, security/perf/UX review, отладка первопричины, неоднозначные компромиссы) → сильная модель; при поддержке — с высоким reasoning effort.
- Непонятно между двумя соседними уровнями → взять **меньший**, повысить при первом сбое.

Механика Cursor: модель за вызов — `--model <id>` (headless) или `/model` (интерактив); список — `cursor-agent models`. Параметризованные модели принимают effort-скобки (напр. `claude-opus-4-8[effort=high]`, `gpt-5.3-codex-high`) — но пиннить так только при явной необходимости; по умолчанию `auto`. Конкретные id (composer / gpt-*-codex / claude-* …) и их тиры меняются от аккаунта и со временем — определять доступное в рантайме (`cursor-agent models`), не зашивать список.

## Маршрутизация — выбирать из доступного

Нет фиксированной таблицы задача→агент. Подбирать подходящего субагента из доступных **сейчас** (Cursor built-ins + кастомные из `~/.cursor/agents`) по их описаниям. Набор меняется от проекта к проекту — не ссылаться на агентов, которых нет.

Неочевидная маршрутизация (не вывести из описаний агентов):
- **Планирование / архитектура / синтез → в главной сессии** (или plan-режиме). Никогда не делегировать *reasoning*. Для фиксации решённого плана в рецензируемый документ — skill `write-plan`; для «что строить» / сравнения вариантов — `research`; для контракта фичи — `write-spec`.
- Security / performance / UX / code review → соответствующий expert-агент, никогда не главная сессия.
- Поиск по коду / «найти X / где используется Y» → Explore / investigator-агент.
- Долгие сборки / тесты / CI → фоновый агент, никогда не блокировать главную сессию.
- Реализация в стеке → специалист по стеку (Kotlin/Compose/Swift и т.д.), когда доступен; иначе general-purpose.
- Skill-first: задача покрыта установленным skill → использовать его вместо прямого агента.
