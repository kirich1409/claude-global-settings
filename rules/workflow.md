# Рекомендуемый рабочий процесс

Семейство плагинов `developer-workflow` — это набор инструментов по требованию, а не принудительный pipeline. Выбирать то, что нужно задаче; последовательность определяет plan mode. Какой инструмент нужен этой конкретной задаче — решает **Шаг 0 (триаж)** ниже, до всего остального.

## Шаг 0: Триаж — классификация входящего

Прежде чем браться за любую задачу или правку — классифицировать её и выбрать **маршрут**. Триаж выполняется до preparation gate и определяет, нужен ли тот вообще. Дерево решений — **дешёвый выход первым**: цель триажа не навесить церемонию на мелочи, а отсечь тривиальное и развести неоднозначную середину (research vs spec vs план vs сразу).

Идти по веткам сверху вниз, остановиться на первой подходящей:

1. **Тривиально → сразу делать.** Однострочное/механическое изменение с очевидным результатом, правка только доков или только конфига без логики. Мимо всех gates (те же исключения, что у `/finalize` и `/acceptance`). Ни research, ни план не запускать.

2. **Вопрос/расследование, не сборка → ответить или исследовать.** Код менять не нужно → ответить напрямую; копание по своей кодовой базе → Explore; широкое многоисточниковое исследование → скилл `research`. Пирамиды нет, пока код не написан ([[task-types]]).

3. **ЧТО строить ещё не решено → решить сначала.** Подход неясен, несколько конкурирующих вариантов, неизвестна осуществимость или незнакомая технология → скилл `research` (research-first, не вопрос пользователю). Требования размыты И фича крупная → `/write-spec` для контракта (требования + AC). Связка допустима: research выясняет варианты → spec фиксирует выбранный.

4. **Слишком крупно для одной единицы работы → сначала декомпозировать.** Если задача не помещается в один план / spec / PR (несколько независимых deliverable, эпик, затрагивает много несвязанных подсистем) — не пытаться удержать её целиком. Разбить на под-задачи (несколько spec или планов, последовательность PR; `/write-spec` штатно обрабатывает «слишком крупное», preparation gate требует декомпозиции — см. ниже), затем **каждую часть прогнать через этот же триаж** заново. Декомпозиция первична: она меняет одну гигантскую неоднозначную задачу на набор задач, каждая из которых сама маршрутизируется в research / spec / план / сразу.

5. **ЧТО решено и помещается в одну единицу → оценить сложность.**
   - Многофайловое, рискованное, нетривиальное либо с несколькими путями реализации → preparation gate → `/write-plan` (или plan mode) до кода.
   - Решено и узко (один-два файла, понятный путь, известный стек) → сразу реализация. Gates ниже (`/finalize`, `/acceptance`) применяются по своим правилам.

6. **Определить тип задачи → [[task-types]].** Фича / багфикс / миграция / бамп / рефакторинг / инфра / UI / перф / исследование — задаёт источник истины, минимальный уровень пирамиды и когда писать тесты. Маршрут из шагов 1–5 плюс тип из этой матрицы вместе определяют поток в разделе «Потоки» ниже.

**Сигналы маршрута** — что во входящем толкает в какую ветку:

| Сигнал | Маршрут |
|---|---|
| «как лучше», «какие варианты», «можно ли», неизвестная осуществимость, незнакомый стек | `research` |
| крупная фича, размытые требования, много затронутых модулей, нужен контракт/AC | `/write-spec` |
| не помещается в один план/PR, несколько независимых deliverable, эпик | сначала декомпозиция → триаж каждой части |
| решено ЧТО, но нетривиально / многофайлово / рискованно | `/write-plan` или plan mode |
| решено ЧТО, узко, понятный путь | сразу реализация |
| однострочник, доки, конфиг, очевидный результат | сразу, мимо gates |
| вопрос без изменения кода | ответ / Explore / `research` |

Триаж — дешёвая операция рассуждения в главной сессии, не отдельный gate с артефактом: его выход — это выбранный маршрут, дальше работает раздел ниже.

## Обязательные gates

**Preparation gate — до любой реализации.** Автономно, в режиме research-first, собрать всё необходимое для построения *и* верификации — три вывода, реально собранных, а не просто названных:
- **Источники истины** (что означает «готово»): spec/AC, before-state baseline, скриншоты/Figma, debug-repro. Активно *производить* их — делать скриншоты, запускать эмулятор, делать снимок baseline до миграции. Что нельзя сделать самостоятельно и что отсутствует → спросить пользователя, назвав, что деградирует без этого. Подробности: [[qa-and-testing]] §6 + [[task-types]] § Before-state baseline. Начинать реализацию без источников истины, необходимых для верификации, нельзя.
- **Источники знаний** (как это построить): доверенные доки/источники по уровням T1–T4 — см. [[external-sources]]. Для нетривиальных фич проактивно искать эталонный код — stack samples (vendor-endorsed) + популярное OSS в той же предметной области — не только API docs; см. [[verify-library-api]] § Reference implementations. Память агента и код проекта устаревают; при пробеле или сомнении — проверить по официальному источнику, не действовать по памяти. Непонимание или тупик → скилл `research` первым, не вопрос к пользователю.
- **Тестируемость + декомпозиция**: оценить, насколько сложно верифицировать изменение, и предложить упрощения заранее (sample/sandbox app, screenshot tests, несколько эмуляторов), чтобы прототип был проверен быстро до прикосновения к реальному приложению; декомпозировать задачу, слишком большую для одного плана. Подробности: [[task-types]] § Test feasibility gate.

Автономность: концентрировать вопросы в этой prep-фазе; после сбора источников — двигаться без лишних уточнений. Стандартное/очевидное решение — применять, не спрашивать. Пропускать prep только для тривиальных случаев, аналогичных gates ниже.

**Quality gate — `/finalize`.** Обязателен после каждой реализации, где писался код — до объявления задачи завершённой. Finalize отвечает за *то, как написан код*: это полный цикл review→fix→simplify, итерирующийся до тех пор, пока не останется находок выше Minor severity, или выходящий с ESCALATE, требующим решения пользователя. `code-reviewer` — один из компонентов, которым управляет цикл; **разовый запуск code-reviewer НЕ закрывает этот gate**: ревью в одиночку оставляет шаги fix и simplify невыполненными. «Код уже отревьюен» не является основанием для пропуска `/finalize`. Исключения: правки только документации, изменения только конфигурации без логики, однострочные механические изменения с очевидным результатом.

**Acceptance gate — `/acceptance`.** Запускается после `/finalize` — до продвижения PR. Верифицирует реализацию против источника истины (spec, test plan, дизайн или behavioral baseline) и запускает runtime checks, включая `manual-tester` для UI surfaces. Два gate'а ортогональны: `/finalize` проверяет *как написан код* (чистота), `/acceptance` проверяет *что делает код* (работает как задумано) — ни один не заменяет другой, оба обязательны. Те же исключения, что и у `/finalize`.

**PR promotion gate — `/create-pr --promote`** (draft → ready for review) требует явного подтверждения пользователя. Открыть draft PR — рутина; продвижение сигнализирует, что задача завершена, и делает PR видимым ревьюерам — это действие с общим состоянием.

## Потоки

**Нетривиальные фичи:**
1. Plan mode → **preparation gate** (выше): собрать + получить источники истины (spec, Figma, AC list, before-state baseline для миграций), подтвердить источники знаний, оценить тестируемость и декомпозировать. Для неизвестного — research-first. Опциональный `/multiexpert-review` для высокорисковых планов, опциональный `/write-spec` когда изменение слишком большое, чтобы удержать в голове, `/write-plan` для фиксации проверяемого плана.
2. Реализовать на feature branch в worktree. Открыть draft PR заранее через `/create-pr --draft`.
3. `/check` → `/finalize` → `/acceptance` → `/create-pr --promote` (требуется подтверждение пользователя) → `/drive-to-merge`.

**Багфиксы:**
1. Plan mode (отладка + фикс в плане). Зафиксировать шаги воспроизведения в `swarm-report/<slug>-debug.md` — это источник истины для `/acceptance`.
2. Написать падающий тест, воспроизводящий баг, затем реализовать фикс (red-green) → `/check` → `/finalize` → `/acceptance` → PR. Регрессионный тест не опционален, если только не применяется feasibility gate — тогда отслеживаемое исключение, никогда молчаливый пропуск (см. [[task-types]] bug-fix row + [[qa-and-testing]] §4). `/write-tests` может создать scaffold теста.

**Исследовательское QA без spec:** вызвать агента `manual-tester` напрямую через Task tool (скилл не нужен).
