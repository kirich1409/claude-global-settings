# Правила QA и тестирования

Решения по тестированию для всего проекта, все стеки.

## 0. Обязательная стратегия тестирования

Каждая задача, изменяющая код, определяет стратегию тестирования на этапе планирования — какие уровни пирамиды применяются, какие инструменты. Пропустить можно только с весомой причиной («код не изменяется, только markdown»); «просто» / «быстрый фикс» / «очевидно» — не принимаются.

### Пирамида верификации

Уровни строго последовательны — каждый требует прохождения предыдущего. Начинать с L0; переход выше требует обоснования.

| Уровень | Название | Описание |
|---|---|---|
| L0 | Build | проект — или только нужная часть (соответствующее приложение/модуль, не всегда весь репо) — компилируется. Без этого идти дальше бессмысленно. |
| L1 | Статический анализ | lint, проверка типов, code review, аудит зависимостей — применяется всегда |
| L2 | Unit-тесты | быстрые, без устройства, чистая логика |
| L3 | UI-тесты | автоматизированные, нужен эмулятор/устройство |
| L4 | E2E-тесты | полный автоматизированный флоу |
| L5 | Ручная верификация | mobile MCP / `manual-tester` на работающем приложении |

**L5 обязателен для:** бампов версии библиотек (даже patch), tech/framework-миграций, изменений infra-слоя (network, storage, auth, DI), любой задачи «не должно влиять на поведение» — проверять в рантайме, не предполагать.

**L5 — закрывать самостоятельно, автономно.** Не «пусть запустит пользователь». Управлять приложением через mobile MCP (`mcp__mobile__*`), если подключён, иначе `adb`/`android` CLI (см. `rules/android-cli.md`) / `manual-tester` на эмуляторе/симуляторе **по умолчанию**; физическое устройство — только когда изменение требует реального железа, которое эмулятор не воспроизводит (biometric HAL, камера, NFC, GPS, sensor fusion). Проверять доступность эмпирически (`adb devices -l`, `emulator -list-avds`, `xcrun simctl list`) — никогда не объявлять L5 нереализуемым из теории; если нужный AVD/образ не установлен, но легко получить — установить и запустить. Собрать и установить APK самостоятельно, управлять флоу, эмулировать ввод. Привлечение пользователя — **крайняя мера** — только при реальных препятствиях: credentials, которые невозможно получить; backend в закрытой/VPN-сети без доступа; поведение, существующее только на физическом железе.

### Стоимость L5 — не жечь turn'ы на скриншоты

Профилирование (`quickai`) показало: прогоны приёмки — самые дорогие задачи и главный источник кружения. Финальная приёмка M3 — **1540 turn'ов при 76 токенах вывода на turn**: один turn = один тап, дальше скриншот и проверка. Дело не в модели, а в способе управления устройством.

Порядок выбора инструмента при работе через mobile MCP — **от дешёвого к дорогому**:

| Что нужно | Чем брать | Порядок цены |
|---|---|---|
| Есть ли элемент, что на экране | `ui(action:'tree', format:'semantic')` | ~60 токенов |
| Только интерактивные элементы | `ui(action:'tree', compact:true)` | ~100 |
| Найти один конкретный элемент | `ui(action:'find')` | ~150 |
| Быстрая визуальная сверка | `screen(action:'capture', preset:'low')` | ~1500 |
| Полный скриншот | `screen(action:'capture')` | ~3000 |

Правила, экономящие turn'ы без потери качества проверки:

- **Дерево вместо скриншота по умолчанию.** Скриншот нужен, когда проверяется именно *визуальное*: раскладка, letterbox, контраст, соответствие макету. Для «нажалось ли», «появился ли пункт», «сменился ли экран» — дерево, оно в 30–50 раз дешевле.
- **`flow(action:'batch')` для последовательностей.** Каждый тап отдельным turn'ом — главный источник кружения; батч сворачивает шаги в один вызов.
- **Подсказки уже включены**: input-действия возвращают diff интерфейса. Скриншот после каждого тапа — антипаттерн, diff уже пришёл.
- **`screen(diff:true)`** после действия — только изменения, а не весь экран заново.
- Не запрашивать дерево и скриншот вместе — выбрать одно.

Исключение, где скриншоты обязательны: сверка с макетом, регрессии раскладки, доказательство «до/после» для визуального дефекта. Там платить полную цену осознанно.

### Захват логов L5 — фильтрация, scoping, редактирование

Когда L5 читает рантайм-логи тестируемого приложения (logcat / `os_log` / server logs) как сигнал верификации, три правила — обязательны:

- **Опираться на детерминированный верификатор, а не текст лога.** pass/fail определяется кодом завершения теста, результатом сборки или screenshot/`assert_visible`. Лог — это *диагностическая гипотеза*, объясняющая **почему** — никогда не единственный сигнал pass/fail (логи зашумлены и нестабильны). Сканирование крешей дополняет детерминированную проверку, не заменяет её.
- **Фильтровать и сужать scope до попадания в контекст.** Никогда не перенаправлять сырые логи в диалог — это переполняет контекст и хоронит сигнал в шуме. Уровень ≥ ERROR для сканирования pass/fail; scope по package/PID на Android (`--pid=$(adb shell pidof -s PKG)`, `get_logs(level="E", package=…)`), по `subsystem` на iOS (`simctl log show --predicate 'subsystem=="<bundleId>"'`); ограничивать объём (`-m N` / `tail` / last-N); большой вывод → файл или `ctx_batch_execute`, агент читает путь, не байты. Паттерны крешей: Android `FATAL EXCEPTION` / `AndroidRuntime: FATAL` / ANR / unhandled NPE; iOS `fault`/`error`.
- **Редактировать секреты до передачи в контекст.** Логи, которые агент *читает*, подчиняются тому же правилу «что никогда не попадает в лог», что и логи, которые он пишет ([[logging]]): никогда не передавать в контекст сырые `.env`, `curl -v`, или заголовки `Authorization`; маскировать `Bearer .*`, `*_TOKEN`, `*_KEY`, PII. OWASP LLM06 — захваченные логи достигают поставщика модели.

### Одноразовые верификационные тесты

Тесты не обязаны быть постоянными. Для подтверждения миграции или единичного/временного поведения на этапе реализации допустимо **написать тест, запустить его (убедиться, что проходит), затем удалить** — верификация без фиксации теста. Отличие от §4: §4 запрещает пропускать или удалять тесты, которые *сломали* (чужое покрытие); одноразовый тест — это леса, написанные и принадлежащие автору. Оставлять тест постоянным, когда поведение заслуживает постоянного покрытия; использовать одноразовый, когда проверка действительно разовая.

## 1. Gate покрытия public API

Изменённый публичный символ должен проверяться тестом. «Публичный» = Kotlin без `@internal`/`private`, Swift `public`/`open`, TS `export`; всё остальное — внутреннее.

**Тривиальное — тест не нужен:** чистые носители данных (`data class`, Swift `struct` только с хранимыми props, TS interfaces, enums, type aliases); builder DSL без логики; типы, реэкспортирующие уже проверенный символ.

**Нет изменения поведения → нет нового теста.** Чистое перемещение/переименование файла, переупаковка, перемещение символа или правка только импортов — это **не** «изменение» символа; существующие тесты + зелёная сборка уже покрывают его. Никогда не добавлять unit-тесты поверх перемещения без логики; это избыточное тестирование, такой же шум, как избыточное редактирование. Gate срабатывает на изменение поведения или сигнатуры, но не на смену местоположения символа.

**Сопоставление тестов (по приоритету):** (1) имя файла `Foo.kt` ↔ `FooTest.kt` / `FooTests.swift` / `Foo.test.ts`; (2) имя символа встречается в любом тестовом файле того же модуля; (3) явная аннотация (`@CoveredBy("...")`). Ничего не нашлось → gate не пройден: написать тест или пометить как тривиальное до прохождения `/check`.

## 2. Фреймворк приоритетов тестов

Классифицировать каждый случай: **P0** критичный для релиза (краш, потеря данных, безопасность, платежи, auth — падение блокирует релиз); **P1** на основе AC (один тест на каждый AC-N из спецификации, названный по этому AC); **P2** happy path (один наиболее распространённый успешный флоу на каждый интерфейс); **P3** граничные случаи (границы, пустые значения, locale/timezone, большой ввод, race conditions). P4 (косметическое/исследовательское) исключён из формальных планов — только исследовательское QA через агента `manual-tester`.

## 3. Облегчённый тест-план для не-UI

Когда выполняются **все три** условия — нет макетов, поверхность API/library/CLI (нет UI для конечного пользователя), нет ревью `ux-expert` в scope — убрать разделы на основе макетов и покрывать только: валидацию ввода (типы, диапазоны, некорректные данные), переходы состояний (ввод → наблюдаемое изменение), пути ошибок (какое исключение/код ошибки, когда). Пропускать viewport / accessibility / visual-regression.

## 4. Автор исправляет сломанные тесты в том же прогоне — нельзя нарушать

Кто сломал существующие тесты — чинит их в том же PR. `@Ignore` / `xit` / `t.Skip` **запрещены** без ссылки на отслеживаемую задачу в аннотации (`@Ignore("flaky on iOS 17 — JIRA-1234")`). Никаких «merge red», никаких «починим позже», никакого `--skip-test-fix`. `/check` — это gate; skip без отслеживаемой задачи — нарушение hook.

## 5. Инфраструктура тестирования — определяется проектом

Конкретный runner, имена задач и команды — **ответственность проекта** — читать из собственных инструкций проекта (`<repo>/CLAUDE.md`) или конфига сборки, а не из универсальной таблицы здесь. Если проект не указывает — определять по маркерным файлам в корне (`build.gradle*` / `Package.swift` / `package.json` / `pyproject.toml` / `Cargo.toml` / `go.mod` / `Makefile`) плюс конфиг сборки — и **блокироваться и спрашивать** там, где догадка будет неверной: схема/назначение Xcode, флаги Python runner, какой модуль владеет изменёнными файлами в монорепо.

## 6. Источник истины верификации

Обязательный результат планирования — определяет «готово», контракт, с которым `/acceptance` сверяется.

| Тип | Использовать когда | Артефакт |
|---|---|---|
| Задача / требования | явные AC или чёткая задача | заметки плана / список AC |
| Спецификация | слишком большая, чтобы удержать в голове; отслеживаемые AC | `/write-spec` → `docs/specs/<slug>-spec.md` |
| Тест-план | структурированные исполняемые случаи | `/generate-test-plan` → `docs/testplans/<slug>-test-plan.md` |
| Дизайн-макеты | визуальные AC для UI/UX | Figma в спецификации `design.figma`, или скриншоты |
| Debug-артефакт | только для багфиксов — шаги воспроизведения являются контрактом | `swarm-report/<slug>-debug.md` |
| Поведенческий baseline | миграция / «не должно влиять на поведение» | захватить до изменений (см. [[task-types]] § Before-state baseline) |

**Поведенческий baseline:** для «не должно влиять на поведение» / «мигрировать без поломок» состояние до изменений И ЕСТЬ истина. Полное определение — что подходит, что нет, shortcut через покрытие тестами — в [[task-types]] § Before-state baseline (единственный источник). Коротко: захватить до любых изменений (скриншоты / сессия `manual-tester` / `e2e-scenario.md`), сохранить в `swarm-report/<slug>-baseline.md`, затем `/acceptance` проверяет соответствие состояния после 1:1. «Должно быть нормально» — не источник истины.

**Отсутствующий источник:** если не существует и создать нереально, задокументировать в плане: предполагаемое поведение (один абзац), почему нет формального источника, какой proxy используется (например, ручной walkthrough vs описание задачи). `/acceptance` Step 1.5 блокируется, когда источник не найден, и предлагает upstream skill; обоснование предоставляет proxy — это не обход gate.
